# 聊天附件批量传输、独立重试与恢复设计

## 目标

在不改变云盘上传、下载行为的前提下，为聊天消息提供最多 9 个混合附件的可靠传输能力：一条消息的全部上传复用同一条 `10087` TCP 连接，附件可独立失败、删除和重试；接收方通过 `10088` 按需下载，每个附件可独立恢复；应用退出或崩溃后保留未完成状态。

## 隔离边界

聊天附件复用只在以下条件同时成立时启用：

- `uploadPurpose == CHAT_ATTACHMENT`
- `connectionReuse == true`

聊天上传请求同时携带 `batchId = clientMsgId`。旧客户端和云盘请求没有这些字段，服务端继续在单文件完成后释放任务并关闭连接。

客户端新增聊天专用的批次会话和持久化模型，不改变云盘的 `TransferTaskManager`、`DirectoryService` 与 `TransferTaskEntity` 业务语义。聊天任务不进入云盘传输队列或传输列表。

## 客户端上传架构

### 消息级会话

新增 `ChatAttachmentUploadSession`。它为一条聊天消息持有一个上传 `SocketManager` 和一个 `FileTransferService`，连接 `10087` 后顺序传输全部物理文件，批次结束时主动关闭。

图片附件由原图、可读预览图和缩略图组成；普通文件只有原文件。每个物理文件拥有独立的：

- `taskId`
- MD5
- 文件大小和本地路径书签
- 断点偏移
- 上传状态
- 服务端 `fileId`

物理文件不并行传输，帧不交错。收到与当前 `taskId` 匹配的终态 ACK 后，才开始下一个文件。

### 状态

批次状态：

- `uploading`
- `partialFailure`
- `readyToSend`
- `sendingMessage`
- `sent`
- `failedToSend`
- `cancelled`

附件状态：

- `waiting`
- `uploading`
- `succeeded`
- `failed`
- `paused`
- `removed`

成功附件保留正式 `fileId`。单个附件失败后，如果连接仍具备完整的帧边界和 ACK 路由能力，会话继续上传后续附件；如果发生断线、协议损坏或 ACK 任务不匹配，则关闭旧连接并建立新连接处理后续附件。

### 消息发送条件

部分失败时不通过 `10086` 发送正式消息。用户可以独立重试失败附件，或删除失败/等待中的附件。所有未删除附件都成功后，客户端关闭 `10087`，用成功附件的正式 `fileId` 生成一次 `MIXED v2` 消息，再通过 `10086` 发送。

若附件已全部上传但消息发送失败，只重试 `10086` 消息发送，不重复上传文件。

## 服务端上传架构

`FileUploadHandler` 将清理逻辑拆成两层：

1. 文件任务清理：关闭当前文件、释放上传许可、删除当前 `taskId` 上下文、清除连接的当前任务绑定并重新平衡限速器。
2. 连接清理：删除解析器、删除通道缓存并在待写数据发送后关闭 Socket。

聊天复用连接中，单文件成功或可恢复的文件级失败只执行文件任务清理；批次完成由客户端主动断开。云盘上传和旧客户端仍连续执行文件任务清理与连接清理。

完整性校验失败、非法帧、ACK 路由异常等可能破坏连接状态的错误会关闭连接。每个文件的元数据帧都重新验证传输令牌和聊天附件权限，不能因复用连接跳过鉴权。

连接始终只允许一个活动物理文件。服务端完成异步校验并排队写入终态 ACK 后，才清除连接当前任务并允许下一份元数据进入；客户端必须等待终态 ACK，因此不会出现两个文件上下文并存。

## 持久化和重启恢复

新增聊天专用 Core Data 实体：

- `ChatAttachmentBatchEntity`：`clientMsgId`、会话对象、文本、引用消息信息、批次状态和错误。
- `ChatAttachmentTransferEntity`：附件标识、父批次、类型、物理文件角色、本地书签、文件信息、MD5、`taskId`、进度、状态和 `fileId`。
- `ChatAttachmentDownloadEntity`：消息和附件标识、`fileId`、保存目标书签、临时路径、进度和状态。

粘贴到聊天框的截图会先以无损 PNG 写入应用管理目录，再创建上传记录，避免只存在于内存。

应用重启后：

- `uploading` 和 `downloading` 转换为 `paused`，保留最后进度。
- 成功上传的物理文件保留 `fileId`，不重新上传。
- 继续上传时复用原 `taskId` 和 MD5，通过 `RESUME_CHECK` 获取服务端真实偏移。
- 如果服务端已经成功入库但客户端未收到 ACK，服务端返回 `complete + fileId`，客户端直接恢复成功状态。
- 尚未通过 `10086` 发送的消息继续显示在原对话位置。
- 已上传但消息发送失败的批次只重试消息发送。

## 接收端按需下载

普通文件不自动下载。附件卡片独立显示未下载、下载中、已暂停、失败、已完成状态，并提供独立下载或重试按钮。

图片为保证聊天内容可读，可自动拉取并缓存缩略图或预览图；原图只在用户打开大图或主动下载时通过 `10088` 拉取。图片预览加载失败只影响当前图片，并显示独立重试入口。

下载任务持久化 `fileId`、保存目标、临时文件、已下载字节、预期大小和状态。应用重启后恢复为已暂停，用户可以继续。一个附件下载失败不改变同消息内其他附件状态。

发送方优先使用仍存在的本地原文件展示；本地文件不可用时回退到 `10088`。

## UI 行为

发送中的每个附件显示自身进度和状态。失败附件提供“重新上传”和“删除”；等待或暂停附件提供“继续”与“删除”。成功附件保持可见，不因其他附件失败回退。

接收端普通文件卡片显示独立下载状态；图片单元格显示预览加载状态和重试入口。批次处于部分失败时，消息气泡保留在对话中，但不会被当作已送达消息同步给接收方。

## 兼容性和失败处理

- 新字段为可选字段，默认 `connectionReuse = false`。
- 非聊天上传保持现有单文件单连接语义。
- 传输令牌逐文件验证。
- `taskId` 用于 ACK 匹配和幂等恢复，`batchId` 只用于日志、诊断和批次关联，不替代 `taskId`。
- 客户端主动关闭、网络中断和进程退出均触发服务端连接清理；未完成任务按现有断点机制冻结。
- 正式消息只引用成功且未删除的附件，服务端继续验证附件所有权、存在性、文件名和大小。

## 验收范围

- 一条含 9 个不同大小和格式附件的消息只建立一个正常的 `10087` 上传连接。
- 图片的原图、预览图、缩略图复用同一连接。
- 中间附件失败后，前后成功附件不丢失，失败附件可独立重试或删除。
- 所有保留附件成功后只发送一次 `MIXED v2` 消息。
- 应用重启后保留上传、下载进度与独立重试状态。
- 接收方普通附件按需下载，失败可独立重试。
- 图片聊天预览可见，原图按需加载，失败可独立重试。
- 云盘上传、下载仍保持原有连接与任务管理行为。

## 验证约束

按用户要求，本次实现不执行编译或自动化测试。仅进行代码静态检查和 `git diff --check`，由用户自行完成运行验证。
