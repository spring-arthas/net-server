# 文本大帧收发可靠性设计

## 目标

修复 10086 文本连接在登录、好友列表和历史消息响应较大时出现的客户端超时问题，同时保持现有协议帧、认证、好友关系和历史消息查询逻辑不变。

## 已确认根因

1. 服务端 `WriteQueueHelper` 在非阻塞写入未完成时，固定向文件 Selector 注册 `OP_WRITE`；文本连接实际注册在文本 Selector，导致剩余数据无法继续发送。
2. macOS 客户端每次 `hasBytesAvailable` 事件只读取一次 4096 字节，大帧未被完整消费时可能长期停留在 TCP 接收队列，业务层无法获得完整帧。

## 修复边界

### 服务端

- 根据 `SocketChannelContext.handlerType` 选择文本或文件 Selector。
- 部分写入后继续保留原有待写队列和帧顺序。
- 队列排空后仍由现有 `WriteEventHandler` 取消 `OP_WRITE`。

### 客户端

- 一次输入事件持续读取当前所有可用字节。
- 读取到的数据继续进入现有 `receiveBuffer`，使用现有 `FrameParser` 提取所有完整帧。
- EOF、暂停读取和连接清理语义保持不变。

## 不在本次改动中的内容

- 不修改登录、好友关系、历史消息 SQL。
- 不调整协议帧类型和业务响应结构。
- 头像 URL 化及在线状态展示作为后续独立改造，避免与本次根因修复捆绑。

## 验证标准

- 服务端文本连接发生部分写入时，文本 SelectionKey 必须注册 `OP_WRITE`。
- 客户端单次事件能读取超过 4096 字节的数据并组装完整帧。
- 现有 Java 单测、Swift 单测和两端构建通过。
- 用户重启两端后可手工验证首次登录、好友列表和历史消息。
