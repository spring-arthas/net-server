# 上传目录按需自愈设计

## 目标

当上传任务选择的数据库目录有效、但对应物理目录缺失时，net-server 在断点检查阶段根据数据库目录层级自动恢复缺失的目录，再继续现有上传流程，避免数据库目录树与文件系统短暂不一致导致任务失败。

## 边界

- 数据库目录层级是唯一可信来源，客户端只提供 `dirId`。
- 不根据客户端路径或文件名猜测并补建数据库目录记录。
- 不修改 chat-storage 上传请求字段、帧类型、断点协议和动态分块协议。
- 仅修复当前上传目标所需的目录链，不在服务启动时全量扫描存储目录。

## 目录链校验

从上传目标 `dirId` 向上查询，直到当前用户根目录。每一级必须满足：

1. 数据库记录存在。
2. `is_file=N`、`is_exist=Y`、`del=N`。
3. `userId` 和 `userName` 与当前已认证上传用户一致。
4. 父子 ID 连续，不允许环形引用。
5. 最终根目录必须是当前用户的有效根目录。

任一级不满足时拒绝上传，不创建物理目录。

## 路径构建与安全

- 使用配置项 `NIO.FILE.BASE.PATH.LINUX.MAC` 或 Windows 对应配置作为存储根路径。
- 使用认证用户名作为用户根目录名称，再按数据库目录链中的 `fileName` 逐级拼接。
- 对配置根目录、用户根目录和目标目录执行规范路径校验。
- 目标路径必须位于配置存储根目录及当前用户根目录内。
- 已存在的路径节点如果不是目录，或存在符号链接导致规范路径越界，则拒绝上传。
- 不信任数据库中可能陈旧的 `file_path` 来决定创建位置。

## 自愈流程

1. 上传断点检查收到 `dirId` 和认证用户身份。
2. 查询并校验完整目录链。
3. 按 `dirId` 获取并发锁，锁内重新校验目标目录。
4. 使用 `Files.createDirectories` 级联创建缺失物理目录。
5. 再次确认目标路径存在、是目录且规范路径未越界。
6. 将目录链中与规范路径不一致的 `file_path` 更新为实际规范路径。
7. 释放锁并返回目标目录路径，继续原有断点检查、META 和 DATA 流程。

目录创建是幂等操作；多个任务同时上传到同一目录时只会产生同一组目录。

## 异常与日志

- 数据库记录无效：返回明确的目录数据错误，不创建文件系统内容。
- 用户不匹配：返回无权使用目标目录。
- 路径越界或符号链接异常：记录安全告警并拒绝上传。
- 物理路径与普通文件冲突：返回目录路径被文件占用。
- 创建失败：返回文件系统目录恢复失败，保留原始异常日志。
- 自动恢复成功：记录 `dirId`、规范目标路径和创建的目录层级数，不记录敏感凭据。

## 并发与一致性

- 使用目标 `dirId` 粒度的锁避免并发重复修复和路径更新竞态。
- 锁对象在操作完成后清理，避免目录数量增长导致内存常驻。
- 文件系统创建成功但数据库路径更新失败时，本次上传失败；后续重试会复用已创建目录并再次更新数据库。
- 不删除任何数据库记录或现有文件系统目录。

## 验收标准

1. 数据库目录链完整、物理父目录和叶子目录均缺失时能够级联恢复并继续上传。
2. 物理目录已存在时保持原有上传行为。
3. 数据库目标不存在、已删除、不是目录或不属于当前用户时拒绝上传。
4. 路径逃出配置根目录或遇到异常符号链接时拒绝上传。
5. 同一目录的两个并发上传任务都能获得同一个有效目标路径。
6. 目录恢复后数据库 `file_path` 与规范物理路径一致。
7. chat-storage 无需修改协议即可重新启动失败任务。
