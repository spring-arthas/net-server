# net-server 项目架构与功能分析

## 1. 项目定位
`net-server` 是一个基于 Java NIO Reactor 模型的服务端，核心目标是通过自定义二进制帧协议提供：

- 用户注册/登录等认证能力
- 聊天与好友关系能力
- 网盘目录/文件管理能力
- 大文件上传下载与断点续传能力
- 动态（类似朋友圈）发布能力

入口类为 `com.alibaba.server.NetServer`，启动后会初始化配置、Spring 容器、Selector 和各业务端口监听线程。

## 2. 技术栈与运行时组成

### 2.1 技术栈
- Java 8
- Java NIO（Selector/Channel）
- Spring（XML 配置，非 Spring Boot）
- MyBatis + 自定义通用 Repository
- Druid 数据源 + MySQL
- Lombok + SLF4J/Log4j
- Maven（shade 打包 fat jar）

### 2.2 运行时核心组件
- `NetServer`：启动入口
- `NioServerContext`：服务上下文，负责启动流程、Selector 管理、连接资源释放
- `CoreServer`：创建 Selector/Acceptor 线程与事件处理链
- `EventHandlerContext`：责任链调度（Accept/Connect/Read/Write）
- `WorkerThreadPool`：每连接一个持久化 `ChannelWorker`，串行处理该连接数据
- `DefaultChannelPipeLine`：每连接内的业务处理管道

## 3. 分层与目录结构

### 3.1 核心目录
- `src/main/java/com/alibaba/server/nio/acceptor`：连接接入线程（按端口）
- `src/main/java/com/alibaba/server/nio/selector`：Selector 事件循环
- `src/main/java/com/alibaba/server/nio/handler/event`：NIO 事件责任链
- `src/main/java/com/alibaba/server/nio/handler/pipe`：连接内业务处理管道
- `src/main/java/com/alibaba/server/nio/service/file/handler`：协议帧业务处理器（文本、上传、下载）
- `src/main/java/com/alibaba/server/nio/repository`：数据持久层与 Service
- `src/main/resources/spring/applicationContext.xml`：IOC、数据源、事务、Mapper 扫描配置
- `src/main/resources/server.properties`：端口、缓冲区、限速、文件根路径等运行配置

### 3.2 业务分层模型
- 网络层：Acceptor + Selector + EventHandler（连接和 IO 事件）
- 传输层：`FrameUploadParser/FrameDownloadParser` 解析自定义帧协议
- 业务处理层：`TextTransmissionHandler`、`FileUploadHandler`、`FileDownloadHandler`
- 领域服务层：`repository/*/service/impl`
- 数据访问层：`BaseMapperRepository` + 各模块 Mapper/DO/Param/DTO

## 5. 核心处理链路

### 5.1 启动链路
1. `NetServer.main`
2. `NioServerContext.startupServerContext`
3. `BasicServer.startupBasicServer`（读取 `server.properties`）
4. `startupIocContainer`（加载 `applicationContext.xml`）
5. `CoreServer.startupCoreServer`（创建 Selector/Acceptor + 事件处理链）

### 5.2 连接与事件链路
1. Acceptor 接受连接并创建 `SocketChannelContext`
2. 注册到对应 Selector（`OP_READ | OP_WRITE`）
3. Selector 产生活跃 `SelectionKey`
4. `EventHandlerContext` 调用责任链（Read/Write）
5. Read 将字节流读取为 `GroupData`，提交 `WorkerThreadPool`
6. `ChannelWorker` 执行 `DefaultChannelPipeLine`
7. 按 `handlerType` 分发到对应处理器：`TEXT` / `UPLOAD` / `DOWNLOAD`

### 5.3 粘包半包处理
- 文本/上传：`FrameUploadParser` 状态机解析（魔数 + 头 + payload）
- 下载：`FrameDownloadParser` 状态机解析
- 解析器按连接缓存（`remoteAddress -> parser`），连接关闭时清理

## 6. 协议设计

### 6.1 帧结构
统一采用 `8字节头 + N字节数据`：
- Magic（2字节）
- Type（1字节）
- Flags（1字节）
- Length（4字节）
- Data（N字节）

### 6.2 主要帧类型
- 用户：注册、登录、改密、登出
- 好友：好友列表、搜索、申请、处理申请、备注修改
- 聊天：发消息、推送、历史查询、已读上报
- 目录：创建、删除、重命名、移动、两层目录树查询
- 文件管理：文件列表、详情、删除
- 文件上传：断点检查、元数据帧、数据帧、结束帧
- 文件下载：下载请求、ACK、数据流、结束
- 动态：`DYNAMIC_CREATE_REQ`

## 7. 功能能力清单

### 7.1 用户与会话
- 注册（支持 base64 头像上传到本地文件系统）
- 登录（绑定连接上下文，加入在线用户映射）
- 修改密码
- 登出（移除在线状态）

### 7.2 好友体系
- 好友模糊搜索（用户名/昵称）
- 发起好友申请
- 查询待处理申请
- 处理申请（同意后建立双向好友关系）
- 好友别名更新
- 好友列表查询（含未读数、最近未读摘要）

### 7.3 聊天体系
- 私聊消息发送
- 在线用户实时推送，离线仅落库
- 历史消息分页查询
- 已读回执（批量更新未读状态）

### 7.4 网盘目录与文件元数据
- 目录树初始化与两层目录查询
- 目录创建/删除/重命名/移动
- 文件列表分页查询
- 文件详情查询
- 文件删除（DB + 文件系统）

### 7.5 文件上传
- 断点探测（`RESUME_CHECK`）
- 全新上传与断点续传
- 上传任务 `file_task` 状态维护（WAIT/UPLOADING/PAUSED 等）
- 全局+单连接令牌桶限速
- 并发上传数控制（Semaphore）

### 7.6 文件下载
- 下载请求合法性校验（DB + 文件系统）
- 支持 `startOffset` 断点下载
- 下载并发许可控制（Semaphore）
- 基于待写队列的流控与 OP_WRITE 协同

### 7.7 动态
- 支持发布动态（文本 + 多图路径）
- 文本长度与图片数量校验
- 持久化到 `user_dynamic` 表

## 8. 数据模型概览

### 8.1 主要业务实体
- 用户：`user`
- 好友关系：`user_friends`
- 好友申请：`user_friend_apply`
- 聊天消息：`user_friend_message`
- 文件目录/文件元数据：`file`
- 文件传输任务：`file_task`
- 用户动态：`user_dynamic`

### 8.2 持久层特点
- 通用 `BaseMapperRepository` + Param/DO 约定
- 部分 SQL 使用 XML（如 `fileMapper.xml`），部分用注解 SQL（如聊天模块）
- Service 层承担主要业务校验与聚合逻辑

## 9. 可靠性与并发控制设计

- 每连接独立 `ChannelWorker`，避免任务对象频繁创建
- 非文件传输处理使用全局锁，文件传输连接可并行
- 连接关闭时统一清理：上传/下载/文本上下文 + Worker + 限速任务
- 上传限速支持单连接与全局双层令牌桶
- 下载侧通过 `pendingWriteQueue` 和 `OP_WRITE` 配合处理背压
- 启动恢复器可将异常中断的上传/下载任务重置为 `PAUSED`

## 10. 当前架构特征总结

### 10.1 优点
- 多端口隔离不同业务流量，模型清晰
- 自定义协议统一文本与文件传输，扩展帧类型成本低
- 目录/文件/好友/聊天/动态能力已形成闭环
- 断点续传、限速、并发控制、任务恢复等工程化能力较完善

### 10.2 当前边界
- 文本通道承载了较多业务（认证+好友+聊天+目录+文件元数据+动态），`TextTransmissionHandler` 体量偏大
- 断点续传独立端口配置存在，但当前启动逻辑中未启用对应 Acceptor/Selector
- 采用 Spring XML + 自研通用仓储抽象，维护门槛高于主流 Spring Boot + MyBatis-Plus 标准范式

---

如果后续要继续演进，建议优先拆分 `TextTransmissionHandler` 为按业务域的子处理器（认证/好友聊天/目录文件元数据/动态），并补充一份“帧协议字段规范文档”给客户端联调使用。
